### Архитектура StackVM

Виртуальная машина StackVM - стековая ВМ с динамической типизацией. Каждый элемент стека занимает 9 байт: 1 байт - тег типа, 8 байт - значение. Теги типов: 0x00 - none, 0x01 - int, 0x02 - bool, 0x03 - char, 0x04 - const (индекс строки).

**Вариант:** 4. Типизация: динамическая. Формат инструкций: стековый. Банки памяти: code and constants, data.

#### Регистры

| Регистр | Размер | Назначение |
| --- | --- | --- |
| IP | 32 бит | Указатель инструкций (instruction pointer) |
| SP | 32 бит | Указатель вершины стека (stack pointer) |
| FP | 32 бит | Указатель фрейма (frame pointer) |
| IO | 32 бит | Регистр выбора порта ввода-вывода |
| RV_tag + RV_val | 8 + 64 бит | Возвращаемое значение (тег + данные) |
| STDIN_reg | 72 бит | Порт ввода (тег + значение) |
| STDOUT_reg | 72 бит | Порт вывода (тег + значение) |

#### Банки памяти

| Банк | Диапазон адресов | Назначение |
| --- | --- | --- |
| _code | 0x00000000 - 0x0000FFFF | Код программы |
| _constants | 0x00010000 - 0x0001FFFF | Константные данные |
| _databank | 0x00020000 - 0x0002FFFF | Изменяемые данные |
| _stack | 0x00030000 - 0x0003FFFF | Стек вызовов и данных |

Все банки памяти имеют ячейку 8 бит, порядок байт little-endian, без требований к выравниванию (granularity = 0).

#### Модель вызова подпрограмм

Вызов подпрограммы (инструкция call) выполняет следующие действия: (1) аргументы предварительно помещаются на стек в обратном порядке, (2) call сохраняет адрес возврата (4 байта) и текущий FP (4 байта), устанавливает FP = SP, (3) enter N выделяет N*9 байт для локальных переменных, (4) ret N восстанавливает FP и IP из стека, снимает N*9 байт аргументов и помещает возвращаемое значение на стек.

Фрейм стека (от старших адресов к младшим): [локальные переменные][FP (4 байта)][адрес возврата (4 байта)][аргумент 0][аргумент 1]...[аргумент N-1]. Адресация аргументов: load_arg S обращается по адресу (FP - 17) - S*9, где 17 = 4 (ret addr) + 4 (saved FP) + 9 (размер одного слота).

### Структуры данных образа программы

```cpp
struct Operand {
    enum Kind { NONE, INDEX, ICONST, LABEL, CONST_IDX };
    Kind kind;
    int64_t intValue;
    std::string strValue;
};

struct Instruction {
    std::string mnemonic;
    Operand operand;
    std::string comment;
};

struct CodeFragment {
    std::string label;
    std::vector<Instruction> instructions;
};

struct FunctionCode {
    std::string name;
    int numArgs, numLocals;
    std::vector<CodeFragment> fragments;
};

struct ProgramImage {
    std::vector<FunctionCode> code;
    std::vector<ConstantEntry> constants;
    std::vector<DataEntry> data;
    std::vector<std::string> externFunctions;
    std::vector<std::string> includeFiles;
    std::string toListing() const;
};
```

Каждая функция состоит из именованных фрагментов (CodeFragment), соответствующих базовым блокам CFG. Первый фрагмент - пролог (метка = имя функции, инструкция enter). Последний - эпилог (exit-блок, инструкция ret). Промежуточные фрагменты соответствуют узлам CFG, метки формируются по шаблону funcName_hint_blockId.

### Алгоритм кодогенерации

Модуль CodeGenerator обходит функции программы. Для каждой функции выполняется: (1) классификация параметров и локальных переменных с назначением слотов, (2) генерация пролога (enter N), (3) обход базовых блоков CFG в порядке топологической сортировки (BFS от entry-блока), (4) генерация эпилога (exit-блок с ret N).

Функции без тела (пустое определение - только сигнатура и end) интерпретируются как внешние (extern) и не генерируют код. Их реализация подключается через --include.

### Генерация кода для базового блока

Для каждой операции в дереве операций блока рекурсивно вызывается emitOperation(). Бинарные и унарные операции транслируются напрямую в соответствующие инструкции StackVM (add, sub, mul, cmp_eq и т.д.). Вызов функции (OP_CALL) помещает аргументы на стек в обратном порядке (справа налево), затем генерирует call. Присваивание (OP_ASSIGN) генерирует вычисление правой части и store_local/store_arg для левой.

Результат каждого выражения-стейтмента (не присваивания) снимается со стека инструкцией pop, чтобы стек не рос. Исключение - последнее выражение функции, обернутое CFG-построителем в неявный возврат (OP_RETURN): его результат остается на стеке и передается в exit-блок.

### Трансляция управляющих конструкций

Управляющие конструкции (if, while, until, repeat, break) уже преобразованы в CFG на этапе задания 2. Кодогенератор работает не с ними, а с базовыми блоками и дугами между ними. Условные переходы генерируются на основе condition блока: вычисляется условие, затем jnz для перехода в true-ветвь и jmp в false-ветвь. Безусловные переходы (unconditionalNext) генерируют jmp к метке следующего блока.

### Обработка возврата из подпрограмм

В языке варианта 4 последнее выражение функции является неявным возвращаемым значением. CFG-построитель оборачивает его в OP_RETURN. Кодогенератор при обработке OP_RETURN генерирует вычисление возвращаемого выражения и jmp в exit-блок функции, где стоит ret N. Для функций без явного return exit-блок дополнительно генерирует push_none.

### Runtime-библиотека ввода-вывода

Функции readInt и writeInt реализованы вручную на ассемблере StackVM (файл runtime.asm), так как они работают с низкоуровневыми инструкциями io_read/io_write, недоступными из языка варианта 4.

readInt: в цикле читает символы через io_read, пропускает разделители (пробел, \n, \r), накапливает цифры по формуле result = result * 10 + (char - 48), возвращает при встрече разделителя после цифр или EOF.

writeInt: если аргумент 0, выводит '0'. Иначе извлекает цифры делением на 10 (mod дает младшую цифру, div сдвигает), складывая на стек. Затем в цикле снимает со стека и выводит через lio 1 + io_write (порт 1 = stdout).

### Особенности описания StackVM

При разработке описания StackVM в нотации целевых архитектур были обнаружены и решены следующие проблемы:

(1) Приоритет операций в выражениях: выражение fp - 17 - slot * 9 интерпретируется эмулятором как fp - (17 - slot * 9) из-за правоассоциативности. Решение: введение промежуточной переменной let off = slot * 9 и явных скобок (fp - 17) - off.

(2) Двойная запись в порт вывода: присваивания stdout_tag и stdout_val по отдельности вызывали двойной вывод символа (эмулятор реагировал на каждую запись). Решение: использование 72-битного представления stdout_full и атомарная запись (tag << 64) | val.

(3) Теги результатов арифметики: результат операций add, sub, mul, div, mod не устанавливал тег типа 0x01 (int), из-за чего результат терял информацию о типе. Решение: в каждой арифметической инструкции явно записывается _stack[sp - 9] = 0x01.

## 5. Результаты тестирования

### Тестовая программа - калькулятор с рекурсивным Фибоначчи

Исходный код на языке варианта 4 (calculator.v4):

```
def readInt() of int
end

def writeInt(n of int)
end

def fn_add(a of int, b of int) of int
    a + b;
end

def fn_sub(a of int, b of int) of int
    a - b;
end

def fn_mul(a of int, b of int) of int
    a * b;
end

def fn_div(a of int, b of int) of int
    a / b;
end

def fn_mod(a of int, b of int) of int
    a % b;
end

def fn_fib(n of int) of int
    if n == 0 then
        0;
    else if n == 1 then
        1;
    else
        fn_fib(n - 1) + fn_fib(n - 2);
end

def calc(op of int, a of int, b of int) of int
    if op == 1 then fn_add(a, b);
    else if op == 2 then fn_sub(a, b);
    else if op == 3 then fn_mul(a, b);
    else if op == 4 then fn_div(a, b);
    else if op == 5 then fn_mod(a, b);
    else if op == 6 then fn_fib(a);
    else 0;
end

def main()
    op = readInt();
    a = readInt();
    b = readInt();
    result = calc(op, a, b);
    writeInt(result);
end
```

### Фрагмент сгенерированного листинга (fn_add, fn_fib, calc, main)

```
[section _code]
_start:
    init_sp 0x30000
    call main
    hlt

; extern readInt
; extern writeInt

fn_add:
    enter 0
    load_arg 0      ; a
    load_arg 1      ; b
    add
    jmp fn_add_exit_1
fn_add_exit_1:
    ret 2

fn_fib:
    enter 0
    load_arg 0      ; n
    push_int 0
    cmp_eq
    jnz fn_fib_if_then_2
    jmp fn_fib_if_else_4
fn_fib_if_then_2:
    push_int 0
    jmp fn_fib_exit_1
fn_fib_if_else_4:
    load_arg 0      ; n
    push_int 1
    cmp_eq
    jnz fn_fib_if_then_5
    jmp fn_fib_if_else_7
fn_fib_if_then_5:
    push_int 1
    jmp fn_fib_exit_1
fn_fib_if_else_7:
    load_arg 0      ; n
    push_int 1
    sub
    call fn_fib
    load_arg 0      ; n
    push_int 2
    sub
    call fn_fib
    add
    jmp fn_fib_exit_1
fn_fib_exit_1:
    ret 1

main:
    enter 4
    call readInt
    store_local 0   ; op
    call readInt
    store_local 1   ; a
    call readInt
    store_local 2   ; b
    load_local 2    ; b
    load_local 1    ; a
    load_local 0    ; op
    call calc
    store_local 3   ; result
    load_local 3    ; result
    call writeInt
    jmp main_exit_1
main_exit_1:
    ret 0
```

## 7. Выводы

Цель задания достигнута: реализована полная цепочка от исходного кода на языке варианта 4 до исполнения на виртуальной машине. Компилятор производит ассемблерный листинг, который корректно ассемблируется и выполняется эмулятором StackVM.

Это упростило реализацию и сделало кодогенератор независимым от конкретного синтаксиса языка.

Стековая архитектура с динамической типизацией существенно упрощает кодогенерацию по сравнению с регистровой: не требуется распределение регистров, все промежуточные значения живут на стеке. Однако это приводит к увеличению количества инструкций и более глубокому стеку при вычислении сложных выражений.

Рекурсивный Фибоначчи (fn_fib) подтвердил корректность механизма вызовов: вложенные рекурсивные вызовы fn_fib(n-1) и fn_fib(n-2) корректно сохраняют и восстанавливают фреймы стека через пару call/ret. Тест fib(10) = 55 порождает 177 вызовов fn_fib, каждый из которых создает и разрушает собственный фрейм - это убедительная проверка адресации аргументов через (FP - 17) - slot * 9.

Отладка описания StackVM выявила несколько неочевидных особенностей работы с нотацией целевых архитектур: правоассоциативность арифметики в выражениях, побочные эффекты при записи во view-отображения регистров, необходимость явного тегирования результатов операций. Все проблемы были решены и задокументированы в описании ВМ.
